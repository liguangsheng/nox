# v0.0.2 设计草案集

本文集中记录早期内部规划中标为"待设计"的语言、模块、运行时、ABI 项。每段都是
一份设计草案：写清问题、决定不做什么、留出实现入口。具体实现批次在落地前
应再次更新对应草案，必要时拆出独立 ADR。

## 阶段 8.3：受限 `match` / `switch`

状态：已完成。

### 问题

Nox 当前用 `if`/`else if` 链处理多分支判断，对整数/字符串的 dispatch 写起来
冗长。希望补一种结构化的"枚举值到代码块"语法，但不引入完整的模式匹配
(rust 风格 destructuring、guard、嵌套模式)。

### 决定

- 关键字暂定 `match`。语法保守：

  ```nox
  match (expr) {
      1 => { ... }
      2 => { ... }
      _ => { ... }
  }
  ```

- 头部表达式必须是 `int` 或 `str`（v0.0.2 切片）。其他类型给静态错误。
- 每个 case 头是一个字面量；没有 OR pattern、没有范围、没有 guard。
- `_` 是必须的默认分支：v0.0.2 不做穷尽性分析，强制写默认分支以避免静默
  fallthrough。
- 不存在 C 风格 fallthrough；每个 case body 是独立块，结束就跳到 `match`
  之后。
- 表达式形式 (`let value = match (...) {...}`) 暂不进入第一批；先做语句形式。
- 重复 case 字面量是静态错误。

### 不做

- 不支持 record/array/map pattern。
- 不支持值绑定（`case Some(x)` 之类）。
- 不引入新枚举类型；`match` 只是 dispatch。
- 不引入 expression-form match。

### 实现入口

1. lexer 加 `match` 关键字。
2. parser 解析 `match (expr) { ... }`，case 是 `<literal> => { ... }` 或
   `_ => { ... }`。
3. type checker 检查头部表达式类型、每个 case 字面量类型与头部一致、`_`
   存在、case 字面量唯一。
4. bytecode：把每个 case 编译为 `JumpIfFalse` 链，等同人工写 `if/else if`，
   不引入新指令。
5. 测试：正向用例（int、str）、负向（重复 case、缺少 `_`、case 类型不匹配
   头部、头部非 int/str）。
6. 文档：在 `language-v0.md` 加一段，更新 `decisions/`。

## 阶段 8.4：可选值与 host/runtime 错误表达

### 问题

Nox 当前没有"可能没有结果"的类型。host/runtime 边界上一些函数自然是
fallible（`env_get`、`read_text`、未来 `parse_int` 等），现在都用诊断终止
脚本。这逼着脚本提前调用 `exists` 之类做防御性检查。需要确定 v0.0.2 是否引入
optional / result 类型，以及在 C ABI 上如何传递。

### 决定

- v0.0.2 暂不引入 `Option<T>` / `Result<T, E>` / `T?` 语法。
- 但是 *接受* 一种约定：host function 可以让自己的返回值表达"成功 + 数据"
  或"失败 + 默认数据"。比如 `read_text_or` host function 可以接受一个
  fallback string，使用方靠 `contains` / `exists` 显式判断后再调用。
- 真正的语言级 optional 推迟到 v0.0.3：那时同时考虑 `T?` 语法、`null` 分支
  分析、`is null` 判定、C ABI 的 NoxCoreOptional 包装。
- v0.0.2 仍允许 host function 通过 `Diagnostic` 终止脚本；这一行为不变。

### 不做

- 不引入 `null` 检查语法。`Value::Null` 仍存在但只在 explicit `null` 字面量
  和明确返回 `null` 的 host function 中出现。
- 不引入 `unwrap` / `?` 操作符。

### 实现入口

v0.0.2 阶段没有代码改动；只更新 host function 命名约定。v0.0.3 启动时优先扩展
ADR `0001`/`0002` 风格的小决策记录，并在新版本里实现完整设计。

## 阶段 9.2：模块命名空间和冲突检测

### 问题

当前 `import "math.nox";` 把模块的所有 export 平铺到当前作用域。重复名字
（同一文件内 `let` 和 import 名相同；两个 import 同名 export）会产生未定义
行为：当前实现简单覆盖，没有诊断。

### 决定

分两阶段推进：

1. **先实现冲突检测**（v0.0.2，已落地）：
   - 模块解析阶段，把当前文件 top-level 声明集合和被 import 的 export 集合
     合并；任何重复都给静态诊断（包含两个来源 specifier）。
   - 重复 import 同一 specifier 不计为冲突（已被去重）。
   - 当前文件内的 `let` 和 `fn` 已经能被 type checker 检测重复定义；扩展
     诊断让它也覆盖 import 引入的名字。

2. **再决定是否引入命名空间语法**（v0.0.3）：
   - 候选语法：`import "math.nox" as math; math.double(21);`
   - 这要求 `record`-like 字段访问能用在模块对象上，且 `math` 是不可重新
     赋值的命名空间。
   - 决定依赖第一步落地后的真实冲突数量；如果冲突已经很少，命名空间语法
     未必紧迫。

### 不做

- v0.0.2 不引入 `as` 重命名。
- v0.0.2 不引入 wildcard / selective import。
- v0.0.2 不实现 re-export。

### 实现入口

1. 模块 resolver 在合并 import surface 时收集所有 (name, source_specifier)
   对。
2. type checker 在 predeclare_functions / collect_records 阶段检测重复名字，
   给出 `name '<n>' redeclared` 诊断，并标注两个来源（当前模块或某个
   import）。
3. CLI/LSP 通过 code 暴露：`module.name-conflict`。
4. 测试：同名 import + let、两 import 之间重名、record 与 let 同名等。

## 阶段 10.3：`nox test`

### 问题

早期内部规划提到测试运行器，但没决定测试语法、退出码和发现规则。需要写明
约定。

### 决定

- 测试文件命名约定：`*_test.nox`，与被测文件并列。
- `nox test [path...]` 行为：
  - 没有参数时，从当前目录向下递归查找 `*_test.nox`。若有 `nox.toml`，
    从 `modules.source_dirs` 出发；否则从 cwd。
  - 给出路径时，逐个加载（文件或目录均可）。
- 测试函数约定：`fn`，无参数，返回 `bool`。命名约定 `fn test_*()`。
  - 返回 `true` 视作通过；返回 `false` 视作失败。
  - 抛出诊断（运行时错误）视作失败，记录诊断 message。
- 输出格式：人类可读 + `--json` 选项。
  - 默认按文件分组打印 `path::name PASS|FAIL`，最后一行 summary。
  - JSON schema：与 `check --json` 类似，包含 `schema`、`tests[]`、`summary`。
- 退出码：所有测试通过 0；任意失败 1；CLI 用法错误 2。
- 不引入测试宏、断言库、setup/teardown；测试函数自己用 `if` + 返回 bool。

### 不做

- 不引入测试框架。
- 不引入并发执行（顺序运行所有测试）。
- 不引入测试过滤的正则；只支持"路径"过滤。

### 实现入口

1. CLI 加 `test` 子命令，沿用 `RuntimePermissions::cli()`。
2. 测试发现：实现一个 walker 收集所有 `*_test.nox`。
3. 测试运行：对每个文件 `eval`，然后查找当前模块 top-level 的 `test_*`
   函数并调用；如果模块顶层有 expression 也要 ok。
4. 设计文档落地后再写第一个测试 fixture（例如 `tests/fixtures/example_test.nox`）。
5. 决策记录：写一份 ADR 描述这个 schema，便于工具消费。

## 阶段 11.2：脚本 `args()` 和环境变量列表

状态：已完成。

### 问题

CLI 当前没办法把命令行参数传给脚本。`env_get` 只能查一个变量，不能列出
所有键。

### 决定

- 引入 `args() -> [str]`，返回 CLI 传给脚本的位置参数列表，不包含 `nox`
  自身和子命令。
  - `nox run script.nox a b c` → `args()` 返回 `["a", "b", "c"]`。
  - Engine 默认 args 为空数组。`nox` runtime 在 `Runtime::with_permissions`
    之外新增 `set_args(Vec<String>)` 用来注入；CLI 在 `nox run` 路径中
    根据剩余 argv 调用。
  - `args()` 不需要新权限：CLI 参数本来就是脚本启动时的入参。
- 增加 `env_list() -> map[str, str]`，返回当前环境变量字典；需要 `environment`
  权限。脚本可以判断有哪些 key。
- 不引入 `env_set` / `env_unset`，避免脚本静默修改宿主环境。

### 不做

- 不引入命令行 flag 解析框架。
- 不暴露 stdin / stdout 写入函数（沿用最终值打印路径）。

### 实现入口

1. `Runtime` 新增参数存储和 `Runtime::set_args` setter。
2. install_stdlib 注册 `args()` host function，从 `Runtime` 复制当前数组。
3. CLI `run` 子命令收集除入口路径外的剩余参数，调用 `set_args`。
4. `env_list` 实现：迭代 `std::env::vars()`，过滤后构造 `Value::Map`。
5. 测试：CLI smoke 传参后脚本打印 args；env_list 返回 PATH 等已知 key。

## 阶段 13.1：`Engine` session / 模块图

### 问题

`Engine` 当前承载所有状态：模块缓存、host functions、heap、budget。把它当
"长生命周期对象"反复用，会出现状态累积；LSP 这种宿主需要打开多份文档时，
不希望 host function 注册被覆盖、heap 在 session 间共享。

### 决定

- 不立刻拆出 `Session`/`ModuleGraph`。先观察真实需求，避免预设抽象。
- 写明现状的兼容契约：
  - `Engine::new()` 创建独立运行时，互不共享状态。
  - 同一 `Engine` 上的多次 `eval` / `check` 顺序执行，模块加载缓存只在
    单次操作内存活（见 9.3）。
  - host function 注册是累积的，重复名字会覆盖；这是 v0.0.x 本地开发阶段的公共行为。
  - heap 在 `Engine` 生命周期内累积，宿主需要在合适的时机调用
    `collect_garbage()`。
- v0.0.3 再评估是否需要 `Session { engine: Arc<Engine>, ... }` 这样的对象，
  让多 session 共享语言核心、各自维护文档/诊断状态。

### 不做

- 不引入 `Arc<Engine>` 或 `&Engine` shared API。
- 不引入 module graph 缓存暴露。
- 不引入 host state 注册的"作用域"概念。

### 实现入口

v0.0.2 阶段无代码改动；只把上面"现状契约"段落补进 `embedding.md` 的兼容
规则。等下面任一信号出现再启动 v0.0.3 设计：

- LSP 单进程开多 workspace，host state / heap 互相干扰。
- 嵌入宿主希望复用 module 解析结果（增量 check）。

## 阶段 13.2：C ABI 复合值访问

状态：已完成。

### 问题

C ABI 已经可以返回 `string`，并通过只读 owning handle 暴露 `array`/`map`/`record`
返回值内容。`function` 仍只暴露 `kind`；C callback 也仍不能接收复合类型作为参数。

### 决定

设计目标：让 C 宿主能读取脚本返回的 array/map/record；暂不支持构造或修改。

引入"轻量 handle"模型：

- 每个复合 value（`Array`/`Map`/`Record`）在 C 边界上表示为一个 opaque
  指针 `NoxCoreArrayHandle*` / `NoxCoreMapHandle*` / `NoxCoreRecordHandle*`。
- handle 由 Nox 拥有，宿主只读。调用 `nox_core_array_free` /
  `nox_core_map_free` / `nox_core_record_free` 释放，必须恰好释放一次。
- 提供只读访问器：
  - `nox_core_array_len(h) -> usize`
  - `nox_core_array_get(h, index, NoxCoreValue *out) -> NoxCoreStatus`
  - `nox_core_map_len(h)` / `nox_core_map_keys(h, NoxCoreValue *out, usize cap, usize *written)`
  - `nox_core_map_get(h, key_cstr, NoxCoreValue *out)`
  - `nox_core_record_field(h, name_cstr, NoxCoreValue *out)`
- `NoxCoreValue` 的 `string_value` / 复合 handle 字段都要在 header 中
  注释为"Nox-owned, must be freed exactly once"。
- iterator API（`nox_core_map_iter_*`）作为后续扩展，不是第一批。

### 不做

- 不支持在 C 端构造 array/map/record 再传回脚本（v0.0.2 边界仍是 scalar）。
- 不支持 mutate 现有 handle。
- 不支持长生命周期的"借用"模式；handle 是 owning。

### 实现入口

1. 在 `crates/nox_core/src/ffi.rs` 引入 handle 类型和 alloc/free 函数。
2. 把 `Value::Array(Rc<Array>)` 转 handle 时把 `Rc` 装进 `Box::into_raw`，
   release 时 `drop`。
3. `NoxCoreValue` 扩展：`array_handle` / `map_handle` / `record_handle`
   字段。eval/callback 返回复合 value 时填这些字段。
4. 在 `nox_core.h` 中维护新枚举值和函数声明；在 ADR 中明确 ABI 兼容窗口
   （添加在文件末尾，不重新排序）。
5. C smoke 增加读取 array/map/record 的示例。

## 计划顺序

按依赖关系，建议实现顺序为：

1. 8.5 `push` 等可变容器内建：不进入 v0.0.2；等可变数组语义成立后再重新评估。
2. 8.4 可选值、13.1 session：放到 v0.0.3 之后，等真实需求出现再启动。
