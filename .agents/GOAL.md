# Nox 目标

Nox 是一个用 Rust 实现的可嵌入脚本引擎和运行时。

项目拆分方式类似 V8 和 Node.js：

- `nox_core`：可嵌入引擎，角色类似 V8。
- `nox`：默认运行时和 CLI，角色类似 Node.js。

长期目标是提供一个小型、快速、宿主友好的生产级脚本引擎，以及一个实用、可发布、可回滚的默认运行时。宿主应用应当能直接嵌入 `nox_core`，终端用户则可以通过 `nox` 在生产环境中运行脚本。

最终发布目标是让 Nox 达到生产级正式发布标准：对外版本、CLI、Rust API、C ABI、诊断 code、权限模型、文档、测试、分发物和回滚流程都具备可审计证据。进入生产级发布时，项目必须没有已知发布阻断风险；所有剩余风险都必须有明确边界、监控方式、降级或回滚方案。这里的“无风险使用”按工程发布口径定义为：没有已知高危缺陷、没有未说明的兼容破坏、没有默认越权能力、没有无法回滚的发布步骤，而不是承诺软件在数学意义上绝对不会失败。

## 产品形态

- 语言：轻量级静态类型脚本语言，源码文件以 `.nox` 结尾。
- 核心引擎：词法分析器、解析器、静态类型检查器、编译器、字节码 VM、垃圾回收值、诊断，以及 Rust/C 嵌入 API。
- 运行时：构建在 `nox_core` 之上的模块、定时器、异步任务、文件系统/网络能力，以及宿主可控权限。
- CLI：`nox` 命令用于运行脚本、检查语法、后续格式化，以及查看字节码。

## 设计原则

- 嵌入优先：宿主应用控制内存、I/O、权限和取消执行。
- 小核心：保持 `nox_core` 独立于系统运行时服务。
- 运行时在上层：`nox` 是默认的内置能力运行时，但不是使用引擎的唯一方式。
- 显式能力：运行时 API 必须被明确授权，不能默认可用。
- 良好诊断：语法、运行时和宿主边界错误应指向有用的源码位置。
- Rust 原生实现：利用 Rust 的所有权和类型，让 VM 与宿主 API 行为可预测。
- 从一开始静态类型：第一版语法就要求声明变量、参数和返回类型，避免后续功能继承动态语言兼容包袱。
- 生产安全优先：默认配置必须保守，文件、网络、环境、定时器和异步任务能力必须可关闭、可审计、可测试。
- 发布可回滚：每次正式发布必须有版本号、CHANGELOG、tag、构建产物、验证记录和回滚策略，不能依赖口头状态。
- 兼容承诺明确：公开 Rust API、C ABI、CLI JSON、诊断 code 和脚本可见行为必须区分稳定、实验和暂缓，不让下游误用半稳定接口。

## 生产级发布标准

项目只有同时满足以下条件，才视为达到生产级正式发布目标：

1. 版本和发布语义统一：Cargo 版本、git tag、CHANGELOG、release checklist、文档示例和分发产物都指向同一个版本。
2. 本地和远端验证通过：`cargo fmt --all --check`、`cargo test --all`、`cargo clippy --all-targets -- -D warnings`、release gate、本地分发 smoke、embedding regression、robustness smoke、benchmark smoke 和 CI 都通过。
3. 安全边界可证明：默认不授予危险能力；文件系统 allowlist、网络、环境变量、定时器、异步任务和 host callback 错误传播都有正向与负向覆盖。
4. API/ABI 可维护：Rust API 和 C ABI 有兼容矩阵，公开函数、enum 数值、handle ownership、错误字符串生命周期和线程/重入边界都有文档和测试。
5. 诊断和工具可自动化：CLI JSON、LSP diagnostics、project check JSON、退出码和机器可读 code 在文档和测试中保持一致。
6. 运行稳定性有证据：递归、循环、容器、模块、malformed source、host callback、取消执行和资源释放路径都有回归测试；高风险解析/VM 路径进入 fuzz 或等价压力测试。
7. 文档面向真实用户：README、语言文档、CLI 文档、runtime 权限、embedding 指南、release checklist 和 rollback 指南足以支持新用户安装、运行、嵌入和排错。
8. 发布可回滚：如果发布后发现严重问题，可以保留历史 tag、发布 hotfix、标记撤回版本，并让下游明确知道应该升级到哪个版本。

任何未满足上述条件的版本都只能标记为本地 checkpoint、alpha、beta 或 release candidate，不能称为生产级正式发布。

## 初始里程碑

1. 初始化 workspace 和 crate 边界。已完成。
2. 定义第一版静态类型 `.nox` 语法切片：字面量、带类型变量、带类型函数、调用、块和基础控制流。已完成。
3. 实现带源码 span 的词法分析器和解析器。已完成。
4. 在字节码编译前加入静态语义/类型检查。已完成。
5. 用字节码编译和 VM 替换解释器。已完成。
6. 在 `nox_core` 中为 Rust 和 C 宿主加入宿主函数注册。已完成。
7. 在 CLI 中加入 `nox run <file.nox>`。已完成。
8. 加入模块加载和最小标准库。已完成。
9. 加入文件系统、网络、定时器和环境访问的运行时能力控制。已完成。
10. 加入测试、样例和稳定的嵌入指南。已完成。

## 第一阶段非目标

- 完整 JavaScript 兼容。
- Node.js 包生态兼容。
- JIT 编译。
- 浏览器 API。
- 在嵌入 API 稳定前提供大型标准库表面。

## 当前仓库布局

- `crates/nox_core`：可嵌入引擎、解析器、VM、值、诊断和 Rust/C 宿主 API。
- `crates/nox`：默认运行时和命令行入口。
- `examples`：示例 `.nox` 脚本。
- `docs`：设计说明和架构文档。
- `docs/embedding.md`：Rust 和 C 嵌入指南。
